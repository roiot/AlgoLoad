# Karpathy-inspired Codex Skill (text-only)

Этот скилл адаптирует ключевые идеи из `forrestchang/andrej-karpathy-skills` под Codex-рабочий процесс, **без копирования/установки кода** из внешнего репозитория.

## Когда применять
- Когда задача неоднозначная и есть риск неверных предположений.
- Когда есть риск «переинженерить» простое изменение.
- Когда важно ограничить diff только нужными строками.
- Когда нужно добиться результата через проверяемые критерии, а не через расплывчатое «сделай лучше».

## Принцип 1 — Think Before Coding
Перед изменениями:
1. Коротко переформулируй задачу в 1–2 предложениях.
2. Явно перечисли допущения.
3. Если есть двусмысленность — предложи 2–3 интерпретации и выбери одну с обоснованием.
4. Если данных не хватает — остановись и запроси уточнение.

## Принцип 2 — Simplicity First
Пиши минимальное решение:
- Не добавляй фичи, о которых не просили.
- Не вводи абстракцию для одноразового кода.
- Не добавляй «на будущее» конфиги/расширения.
- Убирай лишнюю сложность, если то же можно сделать проще.

Проверка: «Сказал бы опытный инженер, что это переусложнено?» Если да — упростить.

## Принцип 3 — Surgical Changes
Изменяй только то, что относится к задаче:
- Не трогай соседние участки «заодно».
- Не делай побочный рефакторинг без запроса.
- Сохраняй стиль существующего кода.
- Если заметил несвязанную проблему — зафиксируй в комментарии/итоге, но не исправляй без команды.

Разрешено удалять только тот «мусор», который возник **из-за твоих изменений** (неиспользуемые импорты/переменные и т.д.).

## Принцип 4 — Goal-Driven Execution
Преобразуй задачу в проверяемые цели:
- «Исправь баг» → «Сделай воспроизводимый сценарий + проверка, что теперь проходит».
- «Добавь валидацию» → «Добавь проверку для невалидных входов + убедись, что тест проходит».

Для многошаговых задач фиксируй мини-план:
1. Шаг
2. Проверка шага
3. Критерий готовности

## Шаблон поведения Codex
Используй как чеклист перед ответом:
- [ ] Явно назвал допущения.
- [ ] Выбрал самый простой путь.
- [ ] Ограничил изменения рамками задачи.
- [ ] Запустил/описал проверку результата.
- [ ] В финале дал краткий отчёт: что изменено и как проверено.

## Формат финального отчёта (рекомендуемый)
- Что сделано (3–5 пунктов)
- Что НЕ делалось специально (чтобы показать границы)
- Какие проверки выполнены
- Риски/что требует уточнения

## Ограничения
- Этот скилл текстовый и концептуальный.
- Не содержит кода, скриптов или файлов из внешнего репозитория.
- Предназначен как локальный набор правил для работы агента.
